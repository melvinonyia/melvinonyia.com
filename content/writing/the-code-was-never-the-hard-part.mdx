title: The code was never the hard part

There are years' worth of conversations I've had with junior engineers that
all reduce to the same arithmetic. The code was hard, they say, and I learned
it, and now I am good at it, and the work should be easier. It usually isn't,
and the reason is the same in almost every case.

The mechanical work was never what made the role hard. The code is already
getting cheaper to write — autocompletion in 2010, snippets in 2015, copilots
in 2022, agentic loops now. What was scarce a decade ago is no longer scarce.
What was scarce then was the willingness to sit with a problem long enough to
understand who it was actually for.

That part has not gotten cheaper. If anything it has gotten more expensive,
because the cycle from "I have a problem" to "I shipped something that doesn't
fit it" has shrunk by an order of magnitude. The mistakes are faster, the
rework is cheaper, the underlying misunderstanding goes unaddressed.

The skill I keep coming back to is small and unglamorous: spend the extra hour
with the practitioner. Read the second paragraph of the ticket, not just the
first. Sit in on the standup of the team you're shipping for. Ask one more
"and then what?" than you think you need. The answer to the question changes
more often than you'd expect.

The product gets better. The work gets easier. Not because the code got
harder to write, but because the human read got cheaper to do.
