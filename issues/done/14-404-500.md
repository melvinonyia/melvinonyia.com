# Styled 404 + 500 with Cmd-K hint

## What to build

Replace the framework default error pages with on-brand 404 and 500 routes. Each renders the standard header and footer, a short message in the site's voice, a link back to the home page, and a visible hint that ⌘K is available. Activating the hint (click or keyboard) opens the command palette so a visitor on a dead URL can recover without leaving the page.

`/404` must be the response for every URL outside the v1 surface — including all old `/blog/*`, `/work/*`, `/links`, `/press`, and `/stack` paths from the previous site.

## Acceptance criteria

- [x] `/404` renders with header, footer, on-brand message, Home link, and visible ⌘K hint
- [x] `/500` renders with header, footer, on-brand message, Home link, and visible ⌘K hint
- [x] Clicking the ⌘K hint opens the command palette
- [x] Every URL outside the v1 surface returns the 404 page with a 404 status *(via Vercel's `404.html` convention; see Caveats for the unverified-until-deploy clause)*
- [x] Head snapshot tests for both error routes
- [ ] Manual verification: visiting `/blog/anything`, `/links`, `/press`, `/stack`, `/some-random-path` all return the on-brand 404 — *deferred to post-deploy, see Caveats*

## Status note (2026-06-15)

Shipped. Design notes:

- **Two view components, one shape.** `src/components/NotFoundView.tsx` and `src/components/ServerErrorView.tsx` differ only in copy: the 404 reads "Off the map." with a "moved during the rebuild" framing; the 500 reads "Something gave way." with a "try reloading" framing. Both render a `<main>` with the standard responsive padding the rest of the site uses, an eyebrow label (`404` / `500`) above an `<h1>`, prose copy, then a two-button CTA cluster: a TanStack `<Link to="/">` styled as the accent button, and a `<button>` styled as the muted button that calls `useOpenCommandPalette()?.()`. Same DOM contract as the Hero CTAs in everything but the link target — visually consistent.
- **Header and footer are inherited from the root route.** Both views are leaf components rendered inside `<RootDocument>` → `<CommandPaletteController>` → `<SiteHeader>` + `<Outlet>` + `<SiteFooter>`. No special-case wrapper. That means the existing nav, the brand link, the ⌘K trigger button (the small chip in the header), and the footer all show on both error views without copy-paste.
- **The ⌘K hint reuses the existing context.** No new state, no new event bus — the hint calls `useOpenCommandPalette()` exactly the way `SiteHeader` does. So both the `⌘+K` keyboard shortcut and the header chip and the new "Search with ⌘K" button on the error views all open the same palette.
- **Head builders.** `src/lib/seo/notFoundHead.ts` and `serverErrorHead.ts` declare a title, description, and `<meta name="robots" content="noindex">`. They're deliberately *not* part of the headContract registry — error routes have no canonical URL, no OG image, no need to be unfurlable. Per-builder unit tests cover the three required fields.
- **Static + client-side coverage.** Two-track wiring so every entry path renders the right view:
  - **`/404` file route** (`src/routes/404.tsx`) — declared so the prerender can produce `dist/client/404/index.html`. A postbuild script (`scripts/postbuild.mjs`) copies that file to `dist/client/404.html`. Vercel's static-hosting convention is to serve `<outputDirectory>/404.html` for any URL that doesn't map to a file, with a 404 status. So `/blog/foo`, `/links`, `/press`, `/stack`, or anything else outside the v1 surface lands on the on-brand 404 with the right status.
  - **`/500` file route** (`src/routes/500.tsx`) — same prerender + copy treatment, even though Vercel doesn't auto-serve `500.html` on static-asset errors. Emitted for direct-navigation symmetry (`example.com/500` shows the page) and so the same file can be referenced by an explicit `vercel.json` rule if/when needed.
  - **`notFoundComponent: NotFoundView`** on the root route — handles client-side router fallthroughs, including the existing `throw notFound()` in `/work/$slug` and `/writing/$slug` loaders (slugs that don't exist in the content collection). The static 404.html only handles initial HTTP requests for unknown URLs; the client-side notFoundComponent handles SPA-navigation-to-unknown.
  - **`errorComponent: ServerErrorView`** on the root route — wraps the whole subtree in TanStack's error boundary. Any uncaught render error inside an Outlet renders the styled 500 view in-place.
- **Postbuild script lives outside the Vite plugin.** I first tried doing the `404/index.html → 404.html` copy in `emitSeoArtifactsPlugin`'s `closeBundle` hook, but `tanstackStart`'s prerender runs as a separate post-Vite step — by the time it writes `404/index.html`, my plugin's `closeBundle` has already fired and seen no file. The fix is a `scripts/postbuild.mjs` chained after `vite build` in the `build` npm script (`vite build && node scripts/postbuild.mjs`). The script is six lines of explicit `copyFileSync`. Far less moving parts than fighting hook ordering across plugin boundaries.

## Caveats

- **"Every URL outside the v1 surface" verification is post-deploy.** I cannot exercise Vercel's "serve 404.html on unmatched paths" behavior locally — that contract is fulfilled by Vercel's hosting layer, not by anything in the static output. The deployment will need a sanity sweep: `curl -i https://melvinonyia.com/blog/foo`, `/links`, `/press`, `/stack`, `/abcxyz` should each return HTTP 404 with the on-brand body. Acceptance criterion checkbox left open. The file artifact is correct on disk; the runtime serve is conditional on Vercel honoring its convention. If for some reason Vercel does *not* fall back to `404.html` (e.g., if a future rewrite rule in `vercel.json` rewrites everything to `index.html`), add `"errorPage": "404.html"` or an explicit `routes`/`rewrites` rule.
- **Vercel doesn't auto-serve `500.html`.** Static hosting can't generate a 500 in normal operation — the 500 view only renders for runtime errors caught by TanStack's `errorComponent` boundary on the client side. The static `500.html` file is there for direct-navigation completeness (`/500` shows the styled page) and as a parking spot for any future Vercel-side configuration that wants to point at it. The acceptance criterion "renders with header, footer, on-brand message, Home link, ⌘K hint" is satisfied by both the file artifact and the client-side errorComponent.
- **The ⌘K hint button on the error views is JS-driven.** A user with JS disabled will see the button but clicking it will be a no-op (the `useOpenCommandPalette()` context returns `null` outside the React tree's hydration). The Home link still works — that's the primary recovery path. The keyboard shortcut similarly requires JS. Adequate for the modern web target; document.
- **Old `/work/*` paths are not specifically redirected.** The issue calls out `/work/*` paths from the previous site that are no longer in the v1 surface. They'll hit the 404 via the unmatched-route fallback like any other unknown URL. If the old work paths had stable URLs you want to preserve (for inbound links), add explicit Vercel `redirects` rules in `vercel.json` mapping old slugs to new ones, or to `/work` if they have no v1 equivalent. Out of scope for this slice; surface for the cutover (issue #17).
- **The route tree generator regenerates `src/routeTree.gen.ts`** on every dev/build run from the file route list. That file is `.gitignore`-d and is regenerated on first dev/build by the `tanstackRouter` plugin. The first `vite build` of this commit pulls a fresh tree; subsequent typechecks work because the file now exists in the working tree. New contributors run `npm run build` (or `npm run dev`) once before `npm run typecheck`.
